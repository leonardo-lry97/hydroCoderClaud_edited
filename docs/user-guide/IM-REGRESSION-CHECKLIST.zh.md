# IM 四通统一回归测试清单

本文档用于手工回归当前四条 IM 通道：

- 微信 iLink
- 飞书
- 钉钉
- 企业微信

目标不是做抽象设计评审，而是验证”当前用户真实会用到的闭环”是否稳定。

## 一、测试前准备

### 通用环境

- Hydro Desktop 已启动
- API 配置可用，能够正常发起 Agent 会话
- 当前桌面端可以正常收发文本与图片

### 微信

- 已完成扫码授权
- 目标用户已发送过至少一条微信消息，已捕获 `contextToken`

### 飞书

- 飞书应用已启用
- 单聊与群聊都已具备可测试环境
- 机器人具备消息、图片、通讯录等本次用到的权限

### 钉钉

- 钉钉应用已启用并已发布
- 机器人 Stream 模式连接正常
- 已开通主动发送所需最小权限：
  - `qyapi_get_department_list`
  - `qyapi_get_department_member`
  - 上传媒体文件
  - 机器人发消息
- 已在桌面端配置 `robotCode`

### 企业微信

- 智能机器人已创建并启用
- Bot ID / Secret 已填入桌面端配置
- 单聊已具备可测试环境
- 如需测试桌面主动发送，需额外完成 wecom-cli 安装/初始化/通讯录授权

## 二、统一回归主清单

### A. IM 主动发起桌面会话

1. 微信用户发送一条文本消息
   - 预期：桌面端出现对应微信会话，消息进入会话
2. 飞书单聊发送一条文本消息
   - 预期：桌面端出现对应飞书会话，消息进入会话
3. 钉钉单聊发送一条文本消息
   - 预期：桌面端出现对应钉钉会话，消息进入会话

### B. IM 文本往返闭环

1. 在三个通道分别发送普通文本
   - 预期：桌面端收到，Agent 正常回复
2. Agent 回复后回到对应 IM
   - 预期：微信、飞书、钉钉都能收到回复
3. 连续追问两轮
   - 预期：上下文连续，不新建额外会话，不串到别的会话

### C. IM 图片到桌面

1. 微信发送一张图片
   - 预期：桌面端收到并正确渲染
2. 飞书发送一张图片
   - 预期：桌面端收到并正确渲染
3. 钉钉发送一张图片
   - 预期：桌面端收到并正确渲染
4. 飞书/钉钉发送图文混合消息
   - 预期：桌面端既收到文本也收到图片，不被当成空消息

### D. 桌面图片回 IM

1. 在桌面端触发读取本地图片文件并回传
   - 预期：飞书、钉钉能收到图片
2. 微信通道验证当前实现边界
   - 预期：如当前能力不支持同等图片回传，应与现状说明一致，不出现错误冒充成功

### E. 历史会话与恢复

1. IM 发起一个会话，聊 1 轮后关闭
2. 再次从同一 IM 对话发消息
   - 预期：若当前无活跃映射，出现历史选择或新建逻辑
3. 选择恢复历史
   - 预期：恢复成功，继续聊天不再重复弹选择
4. 选择新建会话
   - 预期：新会话建立成功，后续继续消息应保持在该新会话

### F. 改名、关闭、命令链路

1. 在当前已连接会话里执行改名命令
   - 预期：成功改名，不提示“没有活跃会话”
2. 关闭当前会话后再次发消息
   - 预期：按该通道的历史/新建规则处理
3. 命令消息不应被当作普通对话正文传给模型
   - 预期：命令只走命令链路，不污染上下文

### G. 来源标记与桌面介入

1. IM 发起的会话在桌面侧来源筛选可见
   - 预期：微信 / 飞书 / 钉钉来源都能筛选
2. 在 IM 会话对应的桌面会话中输入“桌面介入”消息
   - 预期：对应 IM 收到带 `桌面介入>` 前缀的回传文本
3. 不同来源的会话不应误回到其他 IM
   - 预期：只回当前已绑定的 IM 目标

## 三、飞书专项

1. 飞书单聊首次发消息，桌面端无绑定时
   - 预期：按飞书现行逻辑建立或提示选择，不应无故重复弹选择
2. 飞书卡片恢复历史会话
   - 预期：恢复后继续消息不再重复弹卡片
3. 飞书群聊 `@机器人` 普通文本
   - 预期：桌面端正文中不应残留机器人 mention 文本
4. 飞书群聊 `/命令 @机器人`
   - 预期：命令正确执行，机器人 mention 不应污染命令参数
5. 飞书群聊非机器人 mention
   - 预期：不应误剥离其他人的 mention
6. 飞书桌面主动发起到组织成员
   - 预期：目标可收到，回复直接回当前会话

## 四、钉钉专项

1. 钉钉组织成员列表加载
   - 预期：能够列出可发送成员；若失败，能显示真实错误，不应只显示空列表
2. 桌面端主动发一条钉钉给组织成员
   - 预期：目标收到消息
3. 该成员从钉钉回复
   - 预期：直接进入当前桌面会话，不弹历史会话选择
4. 关闭桌面当前会话后，钉钉再回复
   - 预期：按历史/新建规则处理，不应落到错误会话
5. 钉钉图片消息与图文混合消息
   - 预期：桌面端正确解析

## 五、企业微信专项

1. 企业微信单聊发送一条文本消息
   - 预期：桌面端出现对应企业微信会话，消息进入会话
2. 企业微信 Agent 流式回复
   - 预期：回复以流式方式逐段返回，不是整段一次性
3. 企业微信发送一张图片
   - 预期：桌面端收到并正确渲染
4. 企业微信发送图文混合消息
   - 预期：桌面端既收到文本也收到图片
5. `/help` 命令
   - 预期：返回命令列表
6. `/status` 命令
   - 预期：返回当前会话状态
7. `/sessions` 命令
   - 预期：列出活跃会话
8. `/new my-test` 命令
   - 预期：新建会话，返回确认
9. 历史会话选择
   - 预期：有历史时弹出选择菜单，回复数字正确恢复
10. 桌面主动发送到企业微信成员
    - 预期：目标收到 markdown 消息
11. 该成员从企业微信回复
    - 预期：优先回当前绑定的桌面会话
12. 关闭桌面当前会话后，企业微信再回复
    - 预期：按历史/新建规则处理
13. 企业微信图片消息中 Agent 读取的图片回传
    - 预期：图片正确转发到企业微信
14. 桌面介入消息回传企业微信
    - 预期：收到带 `桌面介入>` 前缀的回传文本

## 六、微信专项

1. 已绑定目标从桌面主动发微信
   - 预期：目标收到消息
2. 微信目标回复
   - 预期：优先回当前绑定桌面会话
3. 未捕获目标
   - 预期：不能发送，且失败原因明确
4. 扫码后但未发送第一条消息
   - 预期：仍不能作为有效发送目标

## 七、多通道交叉验证

1. 普通桌面会话先绑定飞书，再绑定微信
   - 预期：当前允许形成多端桥接房间时，三方消息按现行规则互通
2. 普通桌面会话先绑定钉钉，再绑定飞书
   - 预期：当前允许的跨 IM 绑定提示出现一次，确认后生效
3. 已绑定会话切换来源筛选查看
   - 预期：`source` 语义保持首次来源，不因主动发送而改写
4. 企业微信会话绑定后，其他渠道（飞书/钉钉/微信）再绑定
   - 预期：按现行跨 IM 绑定规则提示一次，确认后生效

## 八、当前已知非缺陷项

以下项目前不应按 bug 处理：

- 钉钉/飞书端普通 Q&A 目前默认不是流式逐段返回，而是整段返回（企业微信除外——支持原生流式）
- 桌面介入回 IM 目前也是按结果打包发送，不是流式
- 微信不是组织通讯录直发模型，只能对已扫码授权且已捕获目标发送
- 企业微信桌面主动发送依赖 wecom-cli 初始化完成，未完成时工具栏列表为空属正常

## 九、建议记录方式

每条用例建议记录：

- 测试时间
- 通道
- 操作步骤
- 实际结果
- 是否通过
- 相关日志关键字

最少应保留以下日志关键字：

- 微信：目标捕获、回流绑定、发送失败原因
- 飞书：message_type、mentions、history-choice、bindSessionToTarget
- 钉钉：listTargets、bindSessionToTarget、_ensureSession、history choice
- 企业微信：WSClient、replyStream、downloadFile、uploadMedia、wecom-cli contact
