# 微信通知与双向聊天使用指南

Hydro Desktop 内置了微信 iLink 通道。它既能让桌面端或定时任务主动给微信用户发消息，也能把微信用户的回信带回桌面会话。

## 核心边界

- 只能发送给**已扫码授权且已捕获**的微信目标。
- 不读取微信通讯录。
- 不能由一个扫码用户代发给其任意微信好友。
- 当前支持 **文本 + 图片** 双向；语音、视频、文件暂未实现。

## 第一次配置

1. 打开 **能力管理** → **微信通知**。
2. 确认 **启用微信通知** 开关处于打开状态。
3. 如需调整轮询节奏，可设置 `轮询间隔 (ms)` 与 `轮询超时 (ms)` 并保存。
4. 点击 **生成授权二维码**，显示二维码。
5. 让需要接收消息的微信用户自己扫码并确认授权。
6. 扫码完成后，让该用户再发送一条微信消息。
7. 系统会自动捕获目标，成功后目标会出现在可发送列表中，并自动收到一条绑定成功提示。

说明：

- 设置页顶部状态标签表示的是**桌面侧微信 bridge 运行态**，不是“微信用户是否已经扫码成功”。
- 关闭微信通知开关后，会停止后台轮询，并隐藏聊天工具栏里的微信入口；但已授权账号和已捕获目标仍会保留。
- 这里的扫码用户就是后续可接收消息的目标用户。
- 如果同一用户重复扫码绑定，系统会以最新成功绑定作为首选目标。
- 如果界面暂时没刷新，可点击刷新；通常不再需要手动捕获。

## 从桌面发微信

有三种入口：

1. **能力管理页发送测试消息**
   - 在 **微信通知** 页面选择目标并发送。
2. **Agent 会话工具栏快捷发送**
   - 聊天输入框工具栏点击微信图标，选择目标后发送。
3. **Notebook 对话工具栏快捷发送**
   - Notebook 对话面板同样支持微信快捷发送。

当你从某个桌面会话主动发给微信后，系统会把这个微信目标绑定到当前会话，后续该目标的回信会优先回到这个会话。

## 微信回信如何回到桌面

当前路由规则如下：

- 如果该微信目标已经和某个桌面会话绑定，回信优先进入那个会话。
- 如果当前没有已绑定的桌面会话，系统会创建或恢复一个 `微信会话` 来承接消息。
- 微信回流消息会显示在聊天窗口中，并带有“来自微信”的来源标记。
- 大模型的文本回复会继续同步发回微信端。

这意味着当前已经形成完整闭环：

- 桌面发起
- 微信接收并回复
- 桌面看到回复
- Agent 再答复并回发微信

## 定时任务如何配合使用

定时任务执行会话会注入微信通知 MCP，因此任务完成后可以主动推送结果给已绑定目标。

常见用法：

- 每天定时生成摘要后发给某个微信用户
- 任务失败时向指定目标发送告警
- 长任务结束后发结果通知

前提仍然不变：目标必须先完成扫码授权并被捕获。

## 当前限制

- 微信底层使用 iLink HTTP 长轮询，不是钉钉那种原生 WebSocket 推送，所以时延通常会比钉钉略高。
- 只能给系统已知目标发消息，不能临时输入任意联系人。
- 目标删除只影响后续发送列表，不会自动解绑历史会话记录。
- 当前微信仍保留独立的授权 / 轮询 / 目标捕获结构，不等同于飞书 / 企业微信那种组织级联系人模型。

## 常见问题

### 扫码后为什么还没有目标

通常是因为该微信用户还没有发送第一条消息。先让对方发一条文字消息，系统才能拿到 `contextToken` 并完成捕获。

### 为什么能发给一个人，发给另一个人失败

通常表示第二个人当前不是有效的已捕获目标，或者旧目标上下文已经失效。让该用户重新扫码并再发一条消息即可恢复。

### 微信回信会不会总是新建会话

不会。只要这个目标之前是从某个桌面会话主动发出的，回信会优先回到原会话。只有没有绑定上下文时，才会新建或恢复微信会话。
