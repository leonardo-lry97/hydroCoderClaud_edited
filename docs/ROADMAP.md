## 后续计划：Agent 文件路径稳健性

### 背景

Agent / 工具调用链路在 Windows 文件路径上仍有历史兼容问题，尤其是：

- 模型输出 `/mnt/d/...`、`/d/...`、`d/workspace/...` 等非标准 Windows 变体
- 中文与英文混合文件名在部分模型下被重写，出现额外空格，导致工具读取失败
- 前端展示、点击预览、主进程读文件三层对路径的归一化规则尚未完全统一

### 当前结论

- 工具卡片展示层已收敛到统一渲染逻辑，但这不能解决模型在 `tool_use.input.file_path` 中直接写错路径的问题
- `tool_use` 输入当前按原样存储与回显，路径漂移更可能发生在模型复写字符串时，而不是前端展示层
- `/mnt/<drive>/...` 仍未纳入统一归一化，后续应与 `/d/...` 一并处理

### 计划边界

后续仅考虑实现低风险、全模型通用的路径防错层，不做静默模糊替换：

1. 确定性规范化
   - 统一处理 `/mnt/d/...`、`/d/...`、`d/workspace/...`
   - 去除包裹引号、尾部标点、重复分隔符
   - 统一盘符大小写与 Windows 分隔符
2. 保守诊断
   - 文件工具在路径不存在时，尝试生成同目录唯一候选
   - 默认返回候选信息，不直接替换到另一个文件执行

### 暂不实施

以下方案暂不进入实现，避免误读错误文件：

- 基于模糊匹配自动把“带空格路径”替换成“无空格路径”
- 在多个候选路径存在时静默猜测目标文件
- 仅针对单一模型做特判修复而不抽象为通用机制

## 阶段二：能力生态建设（中期）

### 3. 图片/视频/文档 Skills 编写，更新市场

**目标**：围绕常见办公场景，提供开箱即用的 Skill

| Skill | 能力 | 优先级 |
|-------|------|--------|
| **图片处理** | 截图标注、图片压缩/格式转换、OCR 文字识别 | 高 |
| **视频处理** | 视频摘要、关键帧提取、字幕生成 | 中 |
| **文档生成** | Word/PPT/PDF 模板生成、报告自动排版 | 高 |
| **数据可视化** | CSV/Excel → 图表、数据分析报告 | 中 |

**交付**：每个 Skill 编写完成后发布到 hydroSkills 市场仓库，CapabilityModal 自动可见。

### 4. 插件市场精选

**目标**：筛选和推荐高质量插件，建立市场信任度

| 工作项 | 说明 |

| **使用指引** | 每个精选插件配使用教程和最佳实践 |

## 阶段三：知识与智能（中远期）

### 7. 知识库接入

**目标**：让 Agent 具备领域知识检索能力，回答更准确

#### 方案 A：MCP 知识库（推荐先行）

```
用户提问 → Agent 判断需要检索 → 调用知识库 MCP Tool
  → 向量检索 → 返回相关片段 → Agent 基于上下文回答
```

- 对接现有 RAG 服务（如 Dify / FastGPT / MaxKB）
- 通过 MCP Server 封装检索接口
- **优势**：改动小，复用现有基础设施

#### 方案 B：本地知识库

```
用户指定知识目录 → 本地 embedding → SQLite/向量库存储
  → Agent 对话时自动检索本地库 → 增强回答
```

- 本地 embedding 模型（如 BGE / GTE）
- 本地向量存储（sqlite-vss / LanceDB）
- **优势**：离线可用，数据不出本地
- **挑战**：embedding 模型体积、索引构建性能

#### 推荐路径

先走方案 A（MCP 对接），快速验证价值；再视需求决定是否做方案 B（本地化）。

## 阶段四：自治演进（远期）

### 8. 自管理版本（参考 OpenClaw）

**目标**：升级为能自我管理、自我优化的 AI 软件系统

#### 8a. Hydro Desktop 升级路线

在现有 Electron 架构上增强：

| 能力 | 说明 |
|------|------|
| **自我配置** | Agent 根据用户习惯自动调整配置、推荐 Skill |
| **自我诊断** | 检测异常 → 自动排查 → 给出修复建议或自动修复 |
| **自我更新** | 检测市场新 Skill/Plugin → 推荐安装 → 一键部署 |
| **任务编排** | 复杂任务自动拆解 → 多 Agent 协作 → 结果汇总 |

#### 8b. 独立项目路线（基于 SDK）

基于 Claude Agent SDK 从零构建：

| 维度 | 设计 |
|------|------|
| **架构** | Agent SDK + Web UI（React/Vue） |
| **核心** | 多 Agent 编排引擎、Tool 注册中心、Memory 管理 |
| **自治能力** | 代码审查 → 自动修复 → 自动测试 → 自动部署 |
| **参考** | OpenClaw 的自管理模式、Devin 的任务编排 |

#### 8c. Plugin 安装链路去 CLI 化（已完成，保留为历史记录）

该方向已在当前主线中落地：插件市场增删改查，以及插件安装 / 卸载 / 更新 / 启停，已切换到桌面端主进程内建的 `src/main/plugin-runtime/` 实现，不再依赖 `claude plugin ...` 子命令。

本次落地覆盖：

- 市场源拉取、校验、缓存与刷新
- 插件包下载 / clone / 版本目录管理
- `~/.claude/plugins/` 注册表维护与孤儿目录清理
- `enabledPlugins` 与 marketplace 状态同步
- 多 scope 安装记录下的 user scope 安全更新 / 卸载

当前仍保留的边界：

- Terminal / Agent 会话本身仍依赖 Claude Code CLI
- 旧 `plugin-cli.js` 仅作为废弃兼容层保留，未从仓库物理删除
- marketplace 删除时对 project/local 配置中的历史引用不做额外清理，当前按最小收口接受

#### 8d. 内建 Cron 与桌面端定时任务能力协同（远期）

当前聊天态已通过会话级 `disallowedTools` 避免 Claude Code 内建 `Cron*` 工具与 `hydrodesktop` 自身定时任务管理能力冲突，但这只是短期隔离策略。

当前现状补充：

- `hydrodesktop` 定时任务本身已改为跟随绑定会话 runtime，不再维护任务级模型配置
- 对 `source === 'scheduled'` 会话，定时任务工具是否继续注入由 `allowScheduledSessionScheduleTools` 控制，默认开启
- 普通身份 prompt 不再因为 `scheduled` 来源被单独屏蔽

远期需要补上更细粒度的能力路由与冲突消解，避免“一刀切”禁用内建 Cron 对其他潜在场景造成副作用：

- 区分 Claude Code 内建 `/loop` / `Cron*` 与 `hydrodesktop` 本地定时任务的目标域与数据域
- 基于用户意图、当前会话类型、任务上下文做工具路由，而不是单纯依赖硬编码禁用
- 支持在同一产品内并存两套调度能力，并让模型能解释自己当前操作的是哪一套
- 为冲突场景补充显式确认或能力提示，降低误调用与状态混淆

#### 决策点

| 维度 | 升级现有 (8a) | 独立开发 (8b) |
|------|-------------|-------------|
| 投入 | 中 | 高 |
| 周期 | 1-2 月 | 3-6 月 |
| 风险 | 低（渐进式） | 中（全新架构） |
| 上限 | 受 Electron 架构约束 | 架构自由度高 |

**建议**：先走 8a 验证核心能力，积累经验后再决定是否启动 8b。
