# 水文站点数据检查工作台规则引擎与算法边界设计

> 状态：设计草案  
> 关联需求：`docs/design/hydrology-workbench/hydrology-station-data-workbench-requirements.md`
> 关联主设计：`docs/design/hydrology-workbench/hydrology-station-data-workbench-design.md`
> 关联后端设计：`docs/design/hydrology-workbench/hydrology-station-data-workbench-data-model-and-backend-design.md`

## 1. 文档目标

本文档用于明确“水文站点数据检查工作台”的规则引擎设计与算法边界，重点解决以下问题：

- 规则体系按什么结构组织
- 检查执行顺序如何设计
- 哪些规则应由 Node.js 直接实现
- 哪些能力适合交给 Python 算法 worker
- 规则结果如何输出给异常中心、统计结果和 AI 助手
- 如何控制首版复杂度，避免在算法细节上过度设计

本文档的基本立场是：

- **规则系统是业务判定核心**
- **算法服务是规则系统的支撑能力，不是业务主控**
- **首版先把规则骨架、执行边界、参数模型和结果闭环搭起来**
- **具体检测算法细节在实现阶段逐步细化，不在本轮设计中过早下钻**

## 2. 设计结论概览

### 2.1 规则引擎定位

规则引擎负责：

- 定义“什么叫异常”
- 对站点数据执行确定性检查
- 统一输出结构化命中结果
- 将结果交给异常流转、统计与 AI 解释模块消费

规则引擎不负责：

- 页面交互
- 人工复核状态流转本身
- 语言解释生成
- 底层 Agent Session 管理

### 2.2 算法边界结论

建议按以下方式划分：

#### Node.js 直接实现

- 缺失检查
- 必填字段检查
- 边界值检查
- 时间点合法性检查
- 人工值 vs 视频识别值一致性检查
- 摘录值 vs 平均值误差检查
- 基础变幅阈值检查

#### Python worker 支撑实现

- 毛刺检测
- 突变检测
- 更复杂的时序质量分析
- 更复杂的变化模式判断
- 视频水尺识别

### 2.3 首版策略

首版不追求“算法先进性”，优先保证：

- 规则结构清晰
- 任务链可跑通
- 异常判定可追溯
- 参数可配置
- 后续可以逐步替换算法实现

## 3. 规则体系结构

## 3.1 四层规则分组

建议延续需求文档中的四层结构：

1. 完整性规则
2. 一致性规则
3. 合理性规则
4. 时序质量规则

### 3.2 分层目的

这样拆分的好处是：

- 业务语义清晰
- 页面筛选与异常分类自然对应
- 不同规则类型可以独立演进
- Node / Python 的边界更容易控制

## 4. 当前规则目录建议

## 4.1 水位规则

首版建议明确以下规则编码：

- `W-001` 人工观测值缺失
- `W-002` 人工观测值与视频识别值一致性检查
- `W-003` 0 点摘录值与全量平均值误差检查
- `W-004` 边界值检查
- `W-005` 时序合法性检查
- `W-006` 变幅检查
- `W-007` 毛刺检查
- `W-008` 突变检查

## 4.2 气温规则

首版建议明确以下规则方向：

- `T-001` 数据缺失检查
- `T-002` 边界值检查
- `T-003` 时序合法性检查
- `T-004` 变幅检查
- `T-005` 毛刺检查
- `T-006` 突变检查

说明：

- 气温当前没有明确的第二来源一致性校验，因此不强行设置一致性规则
- 若未来接入外部采集值或传输值，可再补一致性规则组

## 5. 规则对象模型

## 5.1 `RuleDefinition`

规则定义建议包含：

- `ruleCode`
- `ruleName`
- `dataType`
- `ruleCategory`
- `severityDefault`
- `enabled`
- `executorType`
- `parameterSchema`
- `resultSchema`
- `description`

推荐 `executorType`：

- `node_builtin`
- `python_worker`
- `hybrid`

### 5.2 `RuleExecutionContext`

规则执行上下文建议统一抽象，不让每个规则自行拼装数据。

建议至少包含：

- `station`
- `ruleProfile`
- `businessDate`
- `timeRange`
- `dataType`
- `readings`
- `statsResult`
- `extractResult`
- `taskMeta`

### 5.3 `RuleHit`

规则命中输出建议统一为：

- `ruleCode`
- `stationId`
- `dataType`
- `severity`
- `status`
- `timeRangeStart`
- `timeRangeEnd`
- `sourceReadingIds`
- `compareReadingIds`
- `metrics`
- `decisionMessage`
- `suggestedAction`
- `evidenceSummary`

## 6. 规则执行架构

## 6.1 总体执行流程

```text
任务触发
  ↓
读取站点配置与规则参数
  ↓
准备 RuleExecutionContext
  ↓
按规则分组顺序执行
  ↓
统一汇总 RuleHit
  ↓
写入规则结果
  ↓
生成或更新异常
  ↓
供统计页 / 异常中心 / AI 助手消费
```

## 6.2 推荐执行顺序

建议执行顺序如下：

1. 完整性规则
2. 一致性规则
3. 合理性规则
4. 时序质量规则

原因：

- 缺失问题优先暴露
- 一致性问题依赖多来源值先存在
- 合理性规则多为单值检查，成本低
- 时序质量规则通常计算量较大，放后执行更稳妥

## 6.3 执行策略

建议支持两种执行模式：

### 模式 A：同步规则链

适用于：

- 缺失检查
- 边界检查
- 一致性检查
- 摘录误差检查

特点：

- 简单直接
- 结果稳定
- 容易追溯

### 模式 B：异步算法委托

适用于：

- 毛刺检测
- 突变检测
- 更复杂时序质量分析

特点：

- 由 Node 发起
- 等待 Python 返回结构化结果
- 再由 Node 统一落入 `RuleHit`

## 7. Node 侧规则设计

## 7.1 适合 Node 落地的规则特征

以下规则优先放 Node：

- 判断条件明确
- 阈值驱动明显
- 无需复杂数值建模
- 计算成本低
- 结果需要高度可解释

## 7.2 首版 Node 规则范围

### 完整性规则

- 是否缺少人工观测值
- 是否缺少某时段必要值
- 是否缺少统计所需值

### 一致性规则

- 人工观测值与视频识别值偏差是否超容差

### 合理性规则

- 是否越过站点边界值
- 是否落在明显不合理区间

### 基础时序规则

- 时间点是否缺口过大
- 相邻值变化是否超过基础阈值

### 统计 / 摘录类规则

- 摘录值与全量平均值误差是否超阈值

## 7.3 Node 规则实现原则

- 规则实现必须可单元测试
- 不在规则内部直接查数据库
- 不在规则内部直接发通知
- 不在规则内部直接创建异常记录
- 规则只负责“输入上下文，输出命中结果”

## 8. Python 侧算法边界设计

## 8.1 适合交给 Python 的能力特征

以下能力适合交给 Python：

- 计算过程相对复杂
- 后续算法替换概率较高
- 可能引入数值分析库或图像处理库
- 存在更强的工程复用空间

## 8.2 首版放 Python 的能力

建议首版先只把这些能力边界预留好：

- `spike_detection`
- `jump_detection`
- `advanced_amplitude_analysis`
- `water_gauge_recognition`

### 8.3 本轮不下钻的内容

当前不在设计文档中细化：

- 毛刺检测采用何种具体数学方法
- 突变检测采用何种具体模型
- 图像识别采用何种具体视觉模型
- 是否做多模型融合

这些内容应在实现阶段，根据真实数据样本再定。

## 8.4 Python 输出要求

即使算法在 Python 侧实现，返回结果也必须保持业务友好。

建议至少返回：

- `algorithmCode`
- `matched`
- `confidence`
- `metrics`
- `candidateSegments`
- `message`

Node 再将其转换成统一 `RuleHit`。

## 9. 参数与配置设计

## 9.1 参数配置原则

规则参数必须可配置，但不建议首版就做成高度可视化规则编排器。

首版建议：

- 系统级默认参数
- 站点级覆盖参数
- 规则开关可控

## 9.2 推荐参数分类

### 通用参数

- 边界最小值
- 边界最大值
- 时间缺口容忍度
- 缺失判定时段要求

### 一致性参数

- 人工值与视频值允许偏差
- 偏差判定方式

### 摘录参数

- 摘录值与平均值允许误差
- 误差率阈值

### 时序质量参数

- 最大允许变幅
- 毛刺敏感度
- 突变敏感度

说明：

- 时序算法参数只定义“业务可理解”的参数名
- 算法内部更细的技术参数，不必全部暴露到业务层

## 10. 规则结果与异常映射

## 10.1 映射原则

规则命中不一定直接等于一个独立异常。

建议映射逻辑：

- 同时间段、同站点、同类型问题可归并
- 新命中可升级已有异常的严重度
- 已关闭异常再次命中时可重新打开或新建

## 10.2 严重度建议

建议统一为三档：

- `info`
- `warning`
- `critical`

### 10.3 异常分类映射

- 完整性规则 → 缺失类异常
- 一致性规则 → 一致性异常
- 合理性规则 → 合理性异常
- 时序质量规则 → 时序质量异常

## 11. 与统计和摘录模块的关系

## 11.1 统计前检查

统计前应先执行必要规则，至少确认：

- 数据是否齐全
- 数据是否可用于统计

## 11.2 摘录后检查

摘录流程完成后，应立即触发：

- 摘录值 vs 平均值误差规则

### 11.3 结果共用

统计与摘录模块不重复实现判定逻辑，而是复用规则引擎输出。

正确关系应为：

```text
统计 / 摘录产生业务结果
  ↓
规则引擎对结果执行检查
  ↓
异常中心消费检查结果
```

## 12. 与 AI 的边界

## 12.1 规则引擎与 AI 的职责分工

规则引擎负责：

- 判断是否异常
- 给出规则证据
- 输出建议动作

AI 负责：

- 解释为什么命中规则
- 总结多条异常
- 辅助给出人工处理建议
- 生成日报说明

## 12.2 不应交给 AI 的部分

不应把以下内容交给语言模型判定：

- 缺失判断
- 阈值判断
- 摘录误差判定
- 数据边界判定
- 站点规则是否命中

### 12.3 AI 消费规则结果的方式

AI 上下文建议直接引用规则结果：

- `ruleCode`
- `decisionMessage`
- `metrics`
- `evidenceSummary`
- `relatedReadings`

这样可以避免模型脱离业务事实自由发挥。

## 13. 可扩展性设计

## 13.1 后续可扩展方向

后续可逐步扩展：

- 更多水文要素
- 更多来源一致性规则
- 跨站点联动规则
- 规则模板版本管理
- 算法版本追踪

## 13.2 首版明确不做

首版不建议做：

- 可视化拖拽式规则编排器
- 用户自定义任意脚本规则
- 多层级复杂规则依赖图
- AI 自动决定规则阈值

## 14. 首版实施建议

## 14.1 最小可用规则骨架

首版建议先落这几类：

- 缺失检查
- 边界值检查
- 人工值 vs 视频值一致性检查
- 摘录值 vs 平均值误差检查
- 基础变幅检查

## 14.2 Python 能力先做接口占位

首版可以先做到：

- Node 规则链先完整跑通
- Python 算法入口协议先定好
- 具体毛刺 / 突变算法后续再替换接入

这样可以避免项目一开始就被算法细节拖住。

## 14.3 建议的开发顺序

1. 规则定义表与参数模型
2. Node 规则执行骨架
3. RuleHit 持久化
4. 异常映射服务
5. 统计 / 摘录对接
6. Python 算法网关
7. AI 解释消费

## 15. 结论

这套规则引擎设计的重点不在于“先把算法做得多复杂”，而在于先建立一条稳定边界：

- **规则由业务系统主控**
- **Node 负责确定性规则骨架**
- **Python 负责复杂算法能力**
- **AI 只消费规则结果做解释和辅助**

因此，当前最合适的推进方式就是：

**先把规则引擎骨架、规则编码、参数模型、结果结构、Node / Python 边界定清；具体检测算法等到真实实现时再逐项细化。**
