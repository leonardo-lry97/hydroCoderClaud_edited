# 水文站点数据检查工作台设计文档

> 状态：设计草案
> 关联需求：`docs/design/hydrology-workbench/hydrology-station-data-workbench-requirements.md`
> 关联平台设计：`docs/design/standalone-app/agent-platform-technical-design.md`
> 产品形态：Hydro Desktop 内嵌独立业务应用

## 1. 文档目标

本文档用于在业务需求基础上，给出“水文站点数据检查工作台”的骨架设计方案，重点明确：

- 业务应用应采用什么产品结构
- 业务模块与页面如何拆分
- 检查规则与任务流如何组织
- AI 能力嵌入在哪些节点
- 如何复用 Hydro Desktop 底层 Agent Session 能力，而不破坏现有宿主聊天能力

本文档不进入最终代码实现细节，但会给出明确的架构建议和接入边界，作为下一步信息架构设计、模块设计和实现拆解的基础。

## 2. 设计结论概览

### 2.1 产品定位结论

该应用应被设计为：

**Hydro Desktop 内的独立业务工作台，而不是聊天页的附属功能。**

它在业务上是一个完整应用，在技术上是一个内嵌应用，在 AI 能力上是一个复用宿主底层能力的专业场景客户端。

### 2.2 技术定位结论

该应用的运行架构应采用：

- 业务主干独立
- AI 辅助嵌入
- 规则引擎优先
- Agent Session 复用

即：

- 站点管理、数据管理、检查规则、统计、摘录、异常处理是业务主干
- AI 不负责决定业务规则是否命中
- AI 负责异常解释、建议修正、辅助复核、总结输出
- AI 交互通过复用宿主 Agent Session 实现，而不是为该应用重新接入一套 Claude SDK

### 2.3 Agent 能力复用结论

本应用应复用 Desktop 底层 Agent 能力，但复用方式必须满足以下原则：

- 不直接操作原始 `AgentSessionManager`
- 不直接复用宿主聊天窗口的 session
- 通过内嵌应用专属的 access layer / broker 接入
- 每个业务 AI 任务都具备独立 session 所有权

这意味着本应用未来应作为 `embedded client` 使用 Agent Platform，而不是把业务逻辑直接写进现有聊天窗口。

## 3. 总体架构

### 3.1 业务总体架构

```text
站点配置 / 原始数据 / 视频截图
            ↓
        数据接入层
            ↓
      检查与质控引擎
            ↓
      异常中心 / 复核流转
            ↓
      统计与摘录处理层
            ↓
        成果展示层
            ↘
             AI辅助层
```

### 3.2 在 Desktop 中的宿主位置

```text
Hydro Desktop
  ├─ 现有主聊天 / Developer / Notebook
  ├─ 能力管理 / Session 管理 / 定时任务
  └─ 水文站点数据检查工作台（新独立入口）
         ├─ 业务页面
         ├─ 规则任务
         ├─ 异常处理
         └─ AI辅助面板
```

### 3.3 双层架构原则

本应用必须采用双层设计：

#### 第一层：业务主干层

负责：

- 站点
- 数据
- 检查规则
- 统计
- 摘录
- 异常处理
- 成果展示

#### 第二层：AI 辅助层

负责：

- 异常解释
- 修正建议
- 复核辅助
- 汇总结论
- 报告生成

这样可以确保：

- 即使 AI 不可用，业务主干依然成立
- AI 提升效率，但不替代规则判断

## 4. 产品信息架构

### 4.1 入口形态

建议提供独立入口，而不是作为现有聊天页中的一个 Tab。

推荐入口形态：

- 主工作区中的独立应用入口
- 左侧导航或顶部应用门户入口
- 可打开为独立 workspace

不建议：

- 作为“聊天窗口附带工具”
- 作为“右侧小面板功能”

### 4.2 一级导航建议

建议一级导航如下：

1. 站点管理
2. 实时数据
3. 检查任务
4. 异常处理
5. 统计结果
6. 水位摘录
7. 成果展示

### 4.3 页面职责

#### 站点管理

用于管理站点基础信息和规则参数。

核心内容：

- 站点列表
- 站点详情
- 站点状态
- 规则参数配置

#### 实时数据

用于查看水位和气温数据本身。

核心内容：

- 按站点查看数据
- 按时间查看数据
- 查看人工值、视频识别值等来源
- 查看数据趋势与明细

#### 检查任务

用于查看定时执行的检查与统计任务。

核心内容：

- 今日任务
- 已执行任务
- 待执行任务
- 失败任务
- 手动重跑入口

#### 异常处理

这是核心业务页面。

核心内容：

- 异常列表
- 按规则类型过滤
- 查看异常详情
- 人工复核
- 修正确认

#### 统计结果

用于查看气温与水位统计结果。

核心内容：

- 统计结果列表
- 按日查看结果
- 查看统计状态与口径

#### 水位摘录

用于查看水位 0 点摘录结果与误差判断。

核心内容：

- 摘录值
- 平均值
- 误差判断
- 异常处理

#### 成果展示

用于汇总业务结果。

核心内容：

- 统计摘要
- 异常概览
- 站点质量概览
- 图表与导出视图

#### AI 助手面板

AI 助手应作为右侧上下文辅助面板存在，而不是独立一级导航页面。

核心内容：

- 当前异常的 AI 分析
- 当前统计的 AI 总结
- 当前站点的 AI 问答
- 当前页面上下文的快捷分析入口

## 6. 规则引擎设计

### 6.1 规则分层

建议将规则分为四组：

1. 完整性规则
2. 一致性规则
3. 合理性规则
4. 时序质量规则

### 6.2 当前首批规则

#### 水位

- `W-001` 人工观测值缺失
- `W-002` 人工观测值 vs 视频识别值一致性
- `W-003` 摘录值 vs 全量平均值误差检查
- `W-004` 边界值检查
- `W-005` 时序合法性检查
- `W-006` 变幅检查
- `W-007` 毛刺检查
- `W-008` 突变检查

#### 气温

- 边界值检查
- 时序合法性检查
- 变幅检查
- 毛刺检查
- 突变检查

### 6.3 规则输出建议

统一输出结构应包含：

- `ruleType`
- `ruleCode`
- `severity`
- `stationId`
- `dataType`
- `timeRange`
- `sourceValue`
- `compareValue`
- `message`
- `suggestedAction`
- `status`

### 6.4 规则与 AI 的边界

必须明确：

- 规则引擎负责“判定是否异常”
- AI 不负责替代规则判定
- AI 可基于规则结果生成解释和建议

## 7. 任务与调度设计

### 7.1 定时任务类型

本应用建议复用 Hydro Desktop 现有定时任务能力，至少建立以下任务：

- 气温日统计任务
- 水位日统计任务
- 水位 0 点摘录任务
- 异常复查任务
- 日摘要生成任务

### 7.2 时间点

- 气温统计：每天 `20:00`
- 水位统计：每天 `00:00`
- 水位摘录：每天 `00:00`

### 7.3 与现有 Desktop 能力的关系

当前 Desktop 已有：

- 定时任务体系
- Agent Session
- 桌面能力注入
- 微信通知能力

因此本应用建议：

- 先复用定时任务作为触发层
- 业务执行逻辑由本应用模块承担
- AI 仅在需要解释、建议、总结时触发 Agent Session

### 7.4 任务执行链建议

```text
定时触发
  ↓
读取站点数据
  ↓
执行规则引擎
  ↓
生成异常
  ↓
执行统计/摘录
  ↓
写入结果
  ↓
必要时调用 AI 生成解释/结论
```

## 9. 如何复用 Desktop 底层 Agent Session 能力

这是本设计最关键的部分。

### 9.1 复用目标

本应用希望复用的不是“宿主聊天窗口”，而是：

- 底层 Agent Session 能力
- 底层模型配置
- 底层能力注入机制
- 底层消息 / 流式输出 / 交互请求能力

### 9.2 不应采用的方式

本应用不应：

- 直接复用 `host-ui` 的聊天 session
- 直接通过现有聊天窗口消息流承载业务 AI
- 直接调用 `AgentSessionManager` 内部实例而不做隔离

原因：

- 会污染宿主聊天上下文
- 会导致事件串扰
- 不利于业务会话与通用聊天会话分离

### 9.3 推荐复用方式

本应用应被视为一个 `embedded client`，通过未来的 Agent Platform 接入。

推荐形态：

```text
Hydrology Workbench UI
   ↓
Embedded Agent Bridge
   ↓
Agent Broker / Router
   ↓
AgentSessionManager
```

### 9.4 业务 AI Session 模型

建议为业务 AI 引入“上下文化 session”模型。

即：

- 每次 AI 助手分析都附带业务上下文
- 会话归属为 `embed:hydrology-workbench`
- 与宿主普通聊天分离

建议的 session 级别：

#### 页面级 session

适用：

- 当前站点分析
- 当前异常分析

特点：

- 生命周期短
- 强上下文

#### 工作台级 session

适用：

- 今日汇总
- 阶段总结
- 成果整理

特点：

- 生命周期相对长
- 面向汇总型任务

### 9.5 建议的 Agent Bridge API

未来内嵌应用可用如下调用面：

- `window.hydroAgent.createSession()`
- `window.hydroAgent.sendMessage()`
- `window.hydroAgent.onEvent()`
- `window.hydroAgent.respondInteraction()`
- `window.hydroAgent.closeSession()`

对本应用而言，封装层不应让业务代码直接散落调用这些 API，而应通过业务服务统一封装。

例如：

- `analyzeAnomalyWithAI(anomalyId)`
- `summarizeDailyStats(taskId)`
- `suggestCorrectionForReading(readingId)`

### 9.6 业务上下文注入策略

语言 AI 使用 Agent Session 时，应注入业务上下文，而不是让模型自行猜业务环境。

建议注入内容：

- 当前站点基础信息
- 当前数据类型
- 当前时间范围
- 当前异常记录
- 当前规则命中信息
- 当前人工值 / 视频识别值 / 统计值 / 摘录值

这部分可以通过两种方式实现：

#### 方式 A：结构化 prompt 注入

在 `sendMessage()` 前构造统一业务上下文块

#### 方式 B：专用 MCP 工具注入

通过业务专用 MCP，让模型按需查询：

- 站点
- 数据
- 异常
- 统计结果
- 摘录结果

建议：

- 首版采用“结构化上下文 + 少量专用 MCP”混合模式

### 9.7 与 Skill 的关系

前面的 `hydro-agent-app-dev` skill 解决的是“帮助开发者写应用时优先复用 Hydro 平台”的问题。

本应用自身运行时不依赖那个 skill。

本应用运行时依赖的是：

- Embedded Agent Bridge
- Agent Broker
- Agent Session
- 专用 prompt / MCP / 工具上下文

### 9.8 推荐的 Session 归属策略

建议归属如下：

- `clientId = embed:hydrology-workbench`
- 按页面或任务创建子会话
- 不与 `host-ui` 共享会话列表

这符合现有 Agent Platform 设计中的：

- ownerClientId
- clientType
- event routing
- session ownership

## 12. 技术分层建议

### 12.0 技术选型结论

本应用后端建议采用：

- **Node.js 作为统一后端主控**
- **Python 仅作为算法执行服务**

即：

- Node.js 负责业务 API、任务编排、状态流转、平台集成、Agent Session 接入
- Python 负责图像识别、时序质量算法、数值分析等重计算能力

该结论意味着：

- 后端主宿主不需要切换到 Python
- Python 不承担系统 orchestrator 角色
- Python 是被 Node.js 调用的算法 worker / 算法服务

原因：

- Node.js 更适合继续复用现有 Desktop 体系中的配置、定时任务、Agent Session、MCP、通知与宿主集成能力
- Python 更适合承担视频水尺识别、毛刺/突变/变幅算法、复杂数值与图像处理
- 该分层能在不破坏现有主架构的前提下，兼顾业务编排效率和算法能力扩展

因此本应用推荐的执行模式为：

```text
Hydrology Workbench UI
        ↓
Node.js 业务主控服务
        ↓
Python 算法服务（按需调用）
```

Node.js 侧负责：

- 站点管理
- 数据管理
- 规则调度
- 定时任务编排
- 异常状态流转
- AI 会话接入
- 调用 Python 并接收结果

Python 侧负责：

- 视频水尺识别
- 图像处理
- 毛刺检测
- 突变检测
- 复杂时序质量分析
- 后续高级统计算法

首版实现建议：

- 确定性规则先优先落在 Node.js 侧
- 高计算和算法型能力通过预留算法执行接口接入 Python
- 不建议一开始就把整个后端切换到 Python

暂不建议首版就做：

- 全量修正自动化
- 大规模多任务 AI 协同
- 用聊天替代整个业务主流程

### 13.3 第一阶段与平台能力复用建议

首版最实际的复用顺序：

1. 复用 Desktop 定时任务
2. 复用 Embedded Agent Session
3. 复用宿主题快照
4. 后续再扩展专用 MCP

## 14. 风险与约束

### 14.1 业务风险

- 规则阈值未完全明确
- 气温规则仍待细化
- 修正流程权限仍待明确

### 14.2 技术风险

- Agent Platform 目前尚在设计阶段，内嵌复用能力未正式落地
- 若直接绕过平台设计去调现有聊天 session，后续会带来大量技术债

### 14.3 建议

因此当前正确路径应为：

- 先完成业务骨架设计
- 再按 Agent Platform 设计逐步接入 embedded session 能力
- 不要把业务应用直接耦合进当前宿主聊天逻辑

## 15. 总结

“水文站点数据检查工作台”应被设计为：

**一个内嵌在 Hydro Desktop 中的独立业务工作台，业务主干以规则与任务为核心，AI 通过专属 embedded Agent Session 方式提供异常解释、修正建议和结果总结。**

这意味着：

- 该应用的主体不是聊天
- 该应用的 AI 能力不能单独再造一套底层 runtime
- 该应用应通过 Agent Platform 的 embedded client 形态复用底层 Agent Session
- 业务上下文必须结构化注入给 AI，而不是依赖通用聊天上下文

后续若继续推进，下一步应输出：

1. 信息架构与页面草图
2. 核心数据模型设计
3. 规则引擎设计
4. AI 助手交互设计
5. 平台接入清单
