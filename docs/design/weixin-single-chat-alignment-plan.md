# 微信单聊版对齐方案

> 最后更新：2026-06-09
> 状态：已按当前方案落地
> 范围：个人微信，不改钉钉 / 飞书 / 企业微信既有稳定行为

本文档定义“微信如何在不破坏三端一致性前提下，纳入当前统一 IM 模型”的实施方案。

这里的目标不是把微信做成飞书/企业微信那种完整版，而是把微信对齐为：

- 单聊版统一实现
- 保留微信平台能力边界
- 与当前三端在业务判断依据和用户可见现象上尽量一致

---

## 一、设计前提

微信与钉钉、飞书、企业微信存在天然能力差异：

1. 无组织级联系人体系
2. 目标主要通过扫码授权 + 首条入站消息捕获
3. 无稳定群聊闭环需求
4. 主动发送依赖 `contextToken`，上下文可能过期

因此，微信不适合强行对齐为“三端完整版”，但适合纳入“单聊版统一模型”。

---

## 二、对齐目标

### 2.1 必须统一的部分

1. IM 会话对象字段语义统一
2. 单聊持久化身份语义统一
3. 入站建会话现象统一
4. 关闭后再次入站的恢复/历史选择逻辑统一
5. `/status`、`/new`、`/resume`、`/close` 等用户可见行为尽量对齐
6. 桌面主动发送成功后绑定、失败不绑定，继续保持

### 2.2 不强行统一的部分

1. 不补微信群聊
2. 不补组织联系人发现
3. 不要求微信具备三端那种稳定主动发起上下文能力
4. 不改变三端已稳定的共享主链行为

---

## 三、落地前问题

本轮改造前，微信链路与统一模型的主要偏差如下：

1. `WeixinBridge` 仍主要使用自有会话决策逻辑，未接入共享 `ImSessionMapper`
2. 当前只支持 `/help`、`/status`，未接入共享命令行为
3. 微信单聊持久化时，仍把 `accountId` 写入 `agent_conversations.im_chat_id`
4. 这会让微信单聊偏离当前统一语义：

```text
单聊应为：
im_user_id = userId
im_chat_id = ''
```

---

## 四、统一后的目标模型

### 4.1 会话对象

微信入站创建的会话继续统一为：

```js
{
  type: 'chat',
  source: 'im-inbound',
  imChannel: 'weixin'
}
```

### 4.2 数据库 IM 身份

微信单聊统一落库为：

```text
im_channel   = 'weixin'
im_user_id   = targetId
im_chat_id   = ''
im_chat_type = 'p2p'
```

说明：

1. `targetId` 指微信目标注册表中的稳定目标键
2. `accountId` 不再进入 `agent_conversations`
3. `agent_conversations` 只表达“会话身份”，不表达“发送运输信息”

### 4.3 发送运输信息

微信主动发送所需的：

- `accountId`
- `displayName`
- `contextToken`
- `hasContextToken`

不再由 `agent_conversations` 承担，而由 `WeixinNotifyService` 的目标注册表承担。

恢复绑定时：

1. 先从会话表读出 `im_user_id`
2. 再按该 `targetId` 去微信目标注册表查目标
3. 若目标仍存在，则恢复发送能力
4. 若目标不存在或上下文过期，则会话身份仍在，但主动发送不可用，需要重新入站激活

## 五、实施结果

### 第一阶段：身份模型对齐

目标：

1. 微信单聊不再把 `accountId` 写进 `im_chat_id`
2. `getBinding()` 不再依赖会话表中的 `accountId`
3. 改为从微信目标注册表恢复运输信息

改动点：

1. `weixin-bridge.js`
2. `weixin-notify-service.js`（如需补充按 `targetId` 查目标的公开能力）
3. `weixin-bridge.test.js`

验收：

1. 主动绑定成功后，DB 中微信单聊为：
   - `im_user_id = targetId`
   - `im_chat_id = ''`
2. 重启或内存丢失后，只要目标注册表中仍有该 target，就能恢复绑定显示
3. 发送仍只在成功后绑定

状态：已完成

### 第二阶段：接入共享会话决策

目标：

1. 微信入站接入共享 `ImSessionMapper`
2. 单聊历史按 `im_channel + im_user_id` 查询
3. 关闭当前会话后再次入站，进入统一历史选择 / 新建逻辑

改动点：

1. `weixin-bridge.js`
2. 必要时少量共享 helper
3. `weixin-bridge.test.js`

验收：

1. 同一目标再次入站，优先回当前映射
2. 当前会话关闭后，不静默复活旧当前，而是按统一逻辑处理
3. 历史会话选择后，身份持久化仍正确

状态：已完成

### 第三阶段：命令行为对齐

目标：

1. 接入共享命令分发
2. 补齐 `/new`、`/resume`、`/close`
3. `/status` 语义与当前三端一致

验收：

1. `status` 能看到当前 / 激活 / 历史状态
2. `new` 可新建微信单聊会话
3. `resume` 可恢复旧会话
4. `close` 后再次入站不出现微信特例偏差

状态：已完成

### 第四阶段：文档与迁移方案回收

状态：已完成

目标：

1. 更新当前 IM 设计文档
2. 修正数据库迁移文档中的微信迁移规则

修正规则：

旧微信迁移目标应为：

```text
im_channel   = 'weixin'
im_user_id   = staff_id
im_chat_id   = ''
im_chat_type = 'p2p'
```

而不是继续把旧 `conversation_id` 迁入 `im_chat_id`。

## 六、测试场景

### 6.1 入站创建

1. 已扫码并有首条入站消息时，桌面无会话，自动创建微信会话
2. 同一目标再次入站，复用当前会话

### 6.2 主动发送绑定

1. 桌面主动发送成功后绑定
2. 主动发送失败不绑定
3. 已绑定会话只显示当前目标

### 6.3 关闭与恢复

1. 当前会话关闭后，同一目标再次入站进入统一恢复/历史选择
2. `/new` 创建新会话后，同一目标可通过 `/resume` 回到旧会话

### 6.4 上下文过期

1. 目标仍存在但 `contextToken` 失效时，会话身份仍可识别
2. 主动发送时提示重新让对方发一条消息激活

---

## 七、实施边界

本方案的底线是：

1. 不破坏钉钉 / 飞书 / 企业微信现有稳定主链
2. 不为了共享而把微信强行做成群聊/组织联系人模型
3. 统一“业务语义”和“用户可见现象”，保留微信平台运输层差异
