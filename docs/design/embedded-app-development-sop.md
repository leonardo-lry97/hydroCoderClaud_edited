# 内嵌 App 开发 SOP

> 状态：执行流程文档  
> 适用范围：Hydro Desktop 新增一个内嵌 app，从立项到提交的标准操作流程  
> 配套文档：
> - [内嵌 App 设计与实现标准](./embedded-app-development-standard.md)
> - [主进程设计](./main-process.md)
> - [渲染进程设计](./renderer.md)

## 1. 目的

这份 SOP 不是解释架构，而是把“新增一个内嵌 app”拆成可执行步骤，降低返工概率。

适用场景：

- 新做一个类似水文工作台的业务工作台
- 对现有内嵌 app 做结构性扩展
- 需要复用宿主 Agent、工作目录、模型切换、微信发送等系统能力

## 2. 角色分工

建议最少明确这三个责任面：

- 产品/业务：定义业务对象、核心流程、必须支持的 Agent 动作
- 前端/桌面：负责页面、上下文、命令桥、共享宿主接入
- 评审人：确认没有偏离共享复用边界，没有重新造聊天壳或文件壳

如果没有独立评审人，提交前至少对照本 SOP 自查一遍。

## 3. 阶段划分

一个新内嵌 app 的开发按 6 个阶段走：

1. 需求澄清
2. 骨架搭建
3. Agent 接入
4. 业务闭环
5. 测试与文档
6. 提交与验收

## 4. 阶段一：需求澄清

在写代码前，先回答下面问题：

### 4.1 业务对象是什么

至少明确：

- 当前对象是谁
- 当前功能页有哪些
- 当前任务或结果对象有哪些
- 哪些对象需要被 Agent 感知

### 4.2 Agent 需要做什么

只写“和当前页面强相关”的动作，例如：

- 切换功能页
- 选中某个对象
- 打开某条详情
- 刷新当前数据

不要把“理论上以后可能有用”的动作提前塞进命令桥。

### 4.3 不该做什么

提前划清边界：

- 不直接扫工作区猜当前 UI
- 不把业务状态写死在 prompt 常量里
- 不新增一套独立聊天 UI
- 不新增一套独立文件树 UI

### 4.4 输出物

这一阶段结束时，至少产出一页简表：

- app 名称
- `appId`
- 当前业务对象列表
- 当前功能页列表
- 计划提供的受控动作列表
- 是否需要业务专属 MCP 工具

## 5. 阶段二：骨架搭建

### 5.1 建页面目录

建议目录：

```text
src/renderer/pages/<app>/
  ├─ index.html
  ├─ main.js
  ├─ styles.css
  ├─ agent-panel.js
  ├─ state.js
  ├─ views.js
  └─ actions.js
```

### 5.2 注册主进程入口

在 `src/main/embedded-app-registry.js` 注册：

- `id`
- `menuKey`
- `titleKey`
- `labelKey`
- `icon`
- `page`
- `window`

### 5.3 补国际化与构建入口

检查：

- `app.windows.xxx`
- `embeddedApps.xxxTitle`
- Vite 多页面入口是否已注册

### 5.4 验收点

阶段二结束时应满足：

- 能在菜单里看到这个 app
- 能成功打开独立窗口
- 页面空壳已正常加载

## 6. 阶段三：Agent 接入

### 6.1 先接共享宿主

固定做法：

- 页面右侧预留挂载点
- `agent-panel.js` 挂 `EmbeddedAgentPanel`
- 通过 `window.hydroAgent` 建 embedded client

### 6.2 不允许的做法

- 复制一份 `AgentRightPanel`
- 页面里内联写聊天逻辑
- 自己重新发明 `agent:*` 调用层

### 6.3 验收点

阶段三结束时应满足：

- 页面打开后右侧 Agent 正常显示
- 能创建/恢复会话
- `工作台助手` / `工作目录` 两个 tab 都能工作

## 7. 阶段四：业务闭环

### 7.1 上下文闭环

先定义 `getContext()`：

- `title`
- `summary`
- `payload`

然后定义哪些状态变化后必须同步：

- 当前对象变化
- 当前功能页变化
- 当前任务变化
- 当前详情选择变化

### 7.2 命令闭环

再定义 `commandHandler()`：

- 命令名
- 参数
- 页面动作
- 返回结构

推荐第一版只做最少动作：

- `refresh`
- `openTab`
- `selectXxx`
- `openXxxDetail`

### 7.3 能力闭环

先用通用：

- `context_get`
- `command_execute`

只有出现明显业务歧义时，再加专属工具。

### 7.4 验收点

阶段四结束时应满足：

- Agent 能知道当前在看什么
- Agent 能切换关键页面状态
- Agent 不会误判业务语义

## 8. 阶段五：测试与文档

### 8.1 必补测试

至少补这几类：

- 注册/开窗 wiring
- preload/IPC bridge wiring
- 宿主组件复用 wiring
- 上下文同步
- 专属能力注入

### 8.2 必补文档

至少更新：

- 总览文档
- 主进程/渲染进程相关章节
- 内嵌 app 标准文档

如果这个 app 带来了新约束或新套路，也要补到 SOP 或标准文档。

### 8.3 验收点

阶段五结束时应满足：

- 自动化测试能覆盖核心 wiring
- 文档已经能指导下一个人照着做

## 9. 阶段六：提交与验收

### 9.1 提交前检查

- 只 stage 本次相关文件
- 检查工作区里是否有无关脏改动
- 跑 `git diff --check`
- 跑相关测试
- 跑 `gitnexus_detect_changes()` 看影响范围

### 9.2 提交说明建议

提交信息建议包含两层语义：

- app 名称
- 变更类型

例如：

```text
feat(embedded-app): add my-workbench scaffold
docs(embedded-app): align embedded app architecture and sop
refactor(hydrology): reuse shared embedded agent panel
```

### 9.3 人工验收清单

至少逐项确认：

1. 菜单能打开 app
2. 页面布局正常
3. Agent 可连续对话
4. 工作目录可浏览和预览
5. 上下文切换准确
6. 关键受控动作可执行
7. 模型/API Profile 切换正常
8. 文档已对齐

## 10. 时间建议

如果是一个常规业务工作台，建议节奏：

- Day 1：需求澄清 + 页面骨架 + 开窗
- Day 2：Agent 接入 + 上下文闭环
- Day 3：命令桥 + 业务专属能力 + 测试
- Day 4：文档对齐 + 提交 + 验收

如果需求很小，可以压成 1 到 2 天，但顺序不要乱。

## 11. 常见失败模式

最常见的问题不是“功能没写完”，而是边界没守住。

### 11.1 失败模式：复制一套聊天壳

表现：

- 页面里自己写消息列表、输入框、工具栏

后果：

- 模型切换、微信发送、交互请求、消息恢复都会重新踩坑

纠正：

- 回到 `EmbeddedAgentPanel` + `AgentChatTab`

### 11.2 失败模式：上下文靠 prompt 硬写

表现：

- prompt 里写很多“当前对象应该是什么”

后果：

- 一旦页面状态变化，Agent 语义立即过期

纠正：

- 用运行态 `contextProvider()` + `notifyContextChanged()`

### 11.3 失败模式：命令桥暴露太宽

表现：

- 给 Agent 一个“任意执行页面动作”的黑盒入口

后果：

- 页面行为不可预测，难测试，也难审计

纠正：

- 动作显式列举，返回结构化结果

### 11.4 失败模式：本来不需要专属工具，却先做了一堆

表现：

- 还没把通用工具用明白，就先加很多业务专属 MCP

后果：

- 维护成本高，语义分层混乱

纠正：

- 先用 `context_get` / `command_execute` 打最小闭环

## 12. 最终标准

一个内嵌 app 是否达标，最终看这四句话：

- 页面是独立的
- Agent 是复用的
- 上下文是运行态的
- 动作是受控的

如果这四句话里有一条不成立，通常就还不该交付。
