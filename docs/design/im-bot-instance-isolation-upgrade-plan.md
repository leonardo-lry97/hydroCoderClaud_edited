# IM 机器人实例隔离升级方案

> 最后更新：2026-06-11
> 状态：设计中，未落地
> 范围：为同一 IM 通道下的不同机器人实例补充隔离能力，优先覆盖飞书，后续可推广到其他通道

本文档用于定义“同一 IM 通道下，不同机器人实例如何在历史会话、恢复、主动绑定、主动发送上实现隔离”的未来升级方案。

本文档不是当前代码事实说明。当前已实现事实应以：

- [im-session-fields-and-binding.md](./im-session-fields-and-binding.md)
- [im-bridge-refactoring.md](./im-bridge-refactoring.md)

为准。

---

## 一、问题背景

当前统一 IM 身份模型主要基于以下字段：

- `im_channel`
- `im_user_id`
- `im_chat_id`
- `im_chat_type`

这套模型已经足够区分：

1. 不同通道
2. 同一通道下的不同单聊对象
3. 同一通道下的不同群聊对象

但它还不能区分：

1. 同一通道下的机器人 A
2. 同一通道下的机器人 B

因此，一旦仅替换某个通道的机器人配置，而入站消息携带的 `userId / chatId` 与旧机器人时期一致，当前系统会继续命中旧历史会话，并沿用既有 `/resume`、历史选择和绑定逻辑。

这在未来支持“同通道多机器人”或“换机器人后历史隔离”时，会造成以下问题：

1. 新旧机器人历史会话串联
2. `/resume` 候选列表不区分机器人实例
3. 主动绑定无法明确“通过哪个机器人发送”
4. 同通道不同机器人之间的会话归属不清晰

---

## 二、升级目标

本次未来升级的目标不是立刻完成“同通道多机器人产品化”，而是先补齐最小完整能力，使系统具备机器人实例隔离的基础。

### 2.1 必须达到的目标

1. 同一通道下，不同机器人实例的历史会话互相隔离
2. `/resume`、`/status`、历史会话查询仅命中当前机器人实例
3. 入站建会话时，持久化写入当前机器人实例标识
4. 主动绑定与主动发送时，能够明确选择机器人实例
5. 老单实例配置自动迁移，不丢失既有历史

### 2.2 暂不要求一步到位的目标

1. 不要求当前所有通道同步支持多实例并发运行
2. 不要求首轮升级同时完成所有通道的实例列表 UI
3. 不要求一步到位改完全部主动推送交互体验

---

## 三、总体设计结论

本次升级不属于“超级架构重做”，但也不只是“数据库加一个字段”。

如果只解决“换机器人后历史串掉”的问题，核心改动集中在：

1. IM 身份模型新增“机器人实例”维度
2. 历史查询、恢复、绑定以该维度参与匹配
3. 配置模型支持机器人实例列表

如果还要支持主动绑定与主动发送，则还需要：

1. 主动绑定时先选择机器人实例
2. 会话绑定关系里记录实例归属

因此，本次升级建议拆成两个层次理解：

1. 历史与身份隔离层
2. 多实例交互与运行时管理层

---

## 四、身份模型升级

### 4.1 当前身份模型

当前共享 IM 身份语义可概括为：

```text
im_channel + im_user_id + im_chat_id + im_chat_type
```

其中：

- 单聊主要依赖 `im_user_id`
- 群聊主要依赖 `im_chat_id`

### 4.2 升级后的目标模型

建议升级为：

```text
im_channel + im_instance_id + im_user_id + im_chat_id + im_chat_type
```

新增字段说明：

- `im_instance_id`
  本地稳定实例 ID，用于标识“同一通道下的哪个机器人实例”

建议额外保留一个“远端身份键”的设计概念，例如：

- 飞书：可参考 `appId`
- 企业微信：可参考 `botId`
- 钉钉：可参考机器人实例配置主键

但远端身份键不直接代替 `im_instance_id`。原因是：

1. `im_instance_id` 需要本地稳定且可控
2. 将来可能允许同一个远端机器人以不同本地实例存在
3. 将来更适合通过“远端身份键变更”来判断是换密钥还是换机器人

### 4.3 单聊与群聊匹配规则

升级后匹配语义建议为：

单聊：

```text
im_channel + im_instance_id + im_user_id
```

群聊：

```text
im_channel + im_instance_id + im_chat_id
```

这样就可以保证：

1. 同通道不同机器人实例不会互相命中历史
2. 同机器人实例下的既有恢复语义保持不变

## 五、数据库升级设计

### 5.1 需要新增的字段

建议在 `agent_conversations` 中新增：

- `im_instance_id TEXT NULL`

如后续 `im_known_chats` 仍参与多实例缓存，也建议同步新增：

- `im_known_chats.im_instance_id TEXT NULL`

### 5.2 查询与索引语义

未来所有按 IM 身份读取历史的查询，都应从：

```sql
WHERE im_channel = ?
  AND im_user_id = ?
  AND im_chat_id = ?
```

升级为：

```sql
WHERE im_channel = ?
  AND im_instance_id = ?
  AND im_user_id = ?
  AND im_chat_id = ?
```

如果后续需要补索引，也建议按该新语义设计。

### 5.3 老数据兼容迁移

迁移原则：

1. 老单实例用户不能丢历史
2. 旧历史需要统一归入“默认实例”

建议方式：

1. 配置迁移时，为旧单实例配置生成固定默认实例 ID
2. 数据迁移时，将该通道的旧历史统一补齐 `im_instance_id = <default_instance_id>`

例如飞书可迁为：

```text
feishu_default
```

但最终命名不应写死在本文档，可由实现时统一抽象。

## 六、配置模型升级

### 6.1 当前问题

当前每个通道更接近“单对象配置”，例如：

```json
{
  "feishu": {
    "enabled": true,
    "appId": "...",
    "appSecret": "..."
  }
}
```

这无法表达“同通道多个机器人实例”。

### 6.2 目标配置结构

建议升级为实例列表：

```json
{
  "feishu": {
    "instances": [
      {
        "id": "fs_001",
        "name": "飞书机器人A",
        "enabled": true,
        "appId": "...",
        "appSecret": "...",
        "defaultCwd": "",
        "maxHistorySessions": 5
      }
    ]
  }
}
```

字段建议：

- `id`
  本地稳定实例 ID
- `name`
  展示名称
- `enabled`
  是否启用
- `appId / appSecret`
  飞书凭据
- `defaultCwd`
  该实例默认工作目录
- `maxHistorySessions`
  历史会话条数

### 6.3 升级兼容规则

老配置自动迁移为：

1. 创建一条默认实例
2. 将旧 `enabled/appId/appSecret/defaultCwd/maxHistorySessions` 搬入该实例
3. 保留旧行为，不要求用户立刻理解多实例

---

## 七、运行时模型升级

### 7.1 当前运行时模型

当前更接近：

```text
一个通道 = 一个 bridge runtime
```

例如：

- 一个 `FeishuBridge`
- 一份 `sessionMap`
- 一组当前绑定与会话身份

### 7.2 未来运行时模型

建议升级为：

```text
一个通道 = 一个 manager
一个实例 = 一个 bridge runtime
```

例如：

- `FeishuBridgeManager`
- `FeishuBridge(instanceA)`
- `FeishuBridge(instanceB)`

每个实例独立维护：

1. 凭据
2. 连接状态
3. 入站事件处理
4. `sessionMap`
5. 主动发送绑定

### 7.3 mapKey 规则升级

当前 `ImSessionMapper.buildKey()` 语义仅依赖：

- 单聊 `userId`
- 群聊 `chatId`

未来建议改为：

单聊：

```text
${instanceId}:${userId}
```

群聊：

```text
${instanceId}:${chatId}
```

这样内存态映射与持久化态查询才能保持一致。

## 八、主动绑定与主动发送设计

### 8.1 为什么这里必须扩展

如果同通道下只有一个机器人实例，那么主动绑定只需要回答：

1. 发给谁

但一旦有多个机器人实例，就必须回答：

1. 发给谁
2. 通过哪个机器人发

因此，主动绑定不能再只存：

```text
session -> target
```

而要升级为：

```text
session -> instanceId -> target
```

### 8.2 建议的用户交互

当通道下只有 1 个可用实例时：

1. 自动选中该实例
2. UI 尽量不增加额外复杂度

当通道下有多个可用实例时：

1. 先选择机器人实例
2. 再选择联系人/群
3. 会话绑定后锁定该实例

### 8.3 会话绑定约束

建议保持当前共享约束的思路：

1. 同一会话只能绑定同一通道下的一个实例
2. 切换实例前需要先解绑或创建新会话

这样可以继续避免“一个会话跨多个外部联系人/机器人乱发”。

## 九、换机器人配置时的规则

这是本次升级必须明确的高风险场景。

### 9.1 当前风险

当前如果只替换飞书配置，而入站 `senderId/chatId` 仍然一致，系统会继续命中旧历史。

### 9.2 未来规则建议

应明确区分两种情况：

1. 同一机器人，仅凭据变更
2. 新机器人替换旧机器人

建议判断规则：

1. 如果远端身份键未变，只是 `secret` 等敏感信息更新
   视为同一实例，沿用历史
2. 如果远端身份键已变
   视为新机器人，应提示创建新实例并与旧历史隔离

这样可避免“用户以为只是换了配置，实际却把新旧机器人混在一起”。

---

## 十、首轮落地建议范围

为了控制风险，建议首轮不要一步做成“全通道多实例产品”。

### 10.1 首轮建议做的事

1. 设计并落地 `im_instance_id`
2. 完成数据库迁移
3. 飞书配置升级为实例列表
4. 飞书入站、历史查询、`/resume`、绑定恢复带上实例维度
5. 主动绑定时支持实例选择

### 10.2 首轮建议暂缓的事

1. 钉钉、企业微信、微信同步多实例化
2. 所有通道统一的多实例管理页
3. 复杂的实例批量启停与排序能力

---

## 十一、实施顺序建议

建议将来按以下顺序落地：

1. 先完成配置模型设计定稿
2. 再完成数据库字段与迁移方案
3. 再修改共享 IM 身份模型与历史查询
4. 再让飞书 bridge runtime 带实例维度运行
5. 最后补主动绑定/主动发送的实例选择 UI

这样做的好处是：

1. 先把“数据语义”定住
2. 再做“运行时接线”
3. 最后补“用户可见交互”

---

## 十二、风险与收益评估

### 12.1 收益

1. 解决换机器人后历史会话串联问题
2. 为同通道多机器人打下基础
3. 明确主动绑定与会话归属
4. 让未来同通道多实例不再依赖隐式行为

### 12.2 风险

1. 会影响历史查询、`/resume`、绑定恢复等共享逻辑
2. 配置迁移如果处理不好，容易造成老用户历史丢失或隔离错误
3. 主动绑定 UI 会从“选联系人”升级为“先选实例再选联系人”

### 12.3 结论

这项升级不属于“超级架构重做”，但属于：

```text
共享 IM 身份模型升级 + 配置模型升级 + 绑定模型升级
```

如果只看“避免换机器人后历史串掉”，改动并不算巨大。  
如果继续向“同通道多机器人完整产品化”推进，后续还需要 runtime 与 UI 进一步实例化。

---

## 十三、最终结论

当前系统尚未区分“同一通道下的不同机器人实例”，因此：

1. 仅替换机器人配置，不会天然隔离旧历史
2. 是否继续命中旧历史，取决于新入站事件的 `userId/chatId` 是否与旧值一致

未来若要真正解决这一问题，最小完整方案应当是：

1. 为 IM 身份模型新增 `im_instance_id`
2. 为配置模型新增机器人实例列表
3. 为主动绑定与主动发送补充实例选择

该方案应先作为设计预案保留，后续再按阶段实施。
